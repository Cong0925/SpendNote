# 项目概述
# ═══════════════════════════════════════════

SpendNote 是一款微信小程序记账应用，使用微信云开发构建，包含：
- **前端**：微信小程序原生开发（WXML + WXSS + JavaScript）
- **后端**：微信云函数（Node.js）
- **数据库**：微信云数据库（MongoDB）

# 固定执行准则（永久有效，全程不可跳过）
# ═══════════════════════════════════════════

1. 所有回复正文开头固定前缀：老板，

2. 接收用户任意提问/需求后，禁止立刻落地执行、禁止直接输出成品内容；
   第一步：精简复述你的理解，分点拆解用户原始诉求，标注用户隐含需求、遗漏条件、边界盲区、不合理设定；
   第二步：基于用户原始方案，针对性列出 >= 2 套优化方案，分别写明：优化方向、改进优势、适用场景、潜在改动点；
   第三步：主动向用户确认：「选用原方案执行 / 选用优化方案A / 选用优化方案B / 自定义调整需求」；
   只有收到用户明确选择指令后，才按照选定方案正式输出内容；无确认指令，持续停留在方案确认环节。

3. 补充约束：若用户需求信息缺失（参数、用途、环境、使用目标模糊），优先补充提问补齐关键信息，再生成优化方案；禁止脑补缺省条件直接动工。
4. 每次对话后，自动更新 当日修改记录文件
5. **禁止私自提交（强制执行，不可跳过）**：任何情况下，禁止未经用户明确同意执行 `git commit`、`git push`、`git merge`、`git rebase` 等影响版本控制的命令。只有当用户明确说「提交」、「commit」、「push」时，才能执行相应操作。违反此规则将导致严重后果。
   - **触发条件**：用户必须明确说出「提交」、「commit」、「push」等指令
   - **禁止行为**：即使代码修改完成、功能开发完成、bug修复完成，也不得主动提交
   - **执行前确认**：每次执行 git 命令前，必须再次确认用户是否明确指示
   - **违规后果**：违反此规则将被视为严重错误

6. **图标缺失提醒（强制执行，不可跳过）**：当代码中需要使用按钮、图标（icon）、图片资源时，如果发现缺失或不存在，必须：
   - **禁止自行创作**：不能使用文字、emoji、CSS模拟、unicode字符等方式替代缺失的图标
   - **立即提醒开发者**：明确告知需要的图标信息，包括：
     1. 图标名称和用途
     2. 图标文件存放路径
     3. 图标主体颜色（阿里图标库支持颜色设置）
     4. 图标尺寸（单位：px）
   - **图标来源说明**：项目使用阿里图标库（iconfont.cn），图标特点：
     - 透明背景，无背景色
     - 可设置图标主体颜色（如黑色、白色、蓝色等）
     - 格式：PNG
     - 可自定义尺寸（单位：px）
   - **示例提醒格式**：
     ```
     老板，
     检测到以下图标缺失，需要您提供：
     1. 图标名称：xxx-icon
     2. 用途：xxx功能
     3. 路径：/images/xxx/
     4. 主体颜色：黑色/白色/蓝色等
     5. 尺寸：48px × 48px
     请从阿里图标库下载PNG格式后，我再继续开发。
     ```
   - **执行时机**：每次代码开发或修改时，如果涉及图标/图片资源，必须先检查文件是否存在

# 操作指令（按触发条件执行）
# ═══════════════════════════════════════════

**创建今日修改记录**：当用户发送「创建今日修改记录」指令时，执行以下步骤：
1. 读取 `docs/修改记录/` 目录下对应月份下所有文件名，提取最大章节号 N
2. 将模板中所有日期占位符（如 `2026-06-05`）替换为当天日期（格式 `YYYY-MM-DD`）
3. 创建新文件 `docs/修改记录/第N+1章 修改记录M-DD.md`（M-DD 为当天月-日） 到到当前月份的文件夹下

**读取错误截图后清理**：当读取 `docs/error imgs/` 目录下的图片文件后，立即删除该文件，避免图片文件积压。

**错误日志处理（强制执行，不可跳过）**：当需要排查用户错误时，必须遵循以下规则：
1. **跳过已处理**：读取 `docs/错误日志/` 目录时，只读取**未处理**的日志文件（文件名不含 `[已处理]`）
2. **标记已处理**：问题解决后，立即将日志文件重命名，添加 `[已处理]` 标识
   - 重命名格式：`原文件名_[已处理].log`
   - 示例：`crash_20260615_143025.log` → `crash_20260615_143025_[已处理].log`
3. **执行时机**：每次读取错误日志进行分析后，无论是否解决问题，都必须标记为已处理
4. **避免重复**：下次排查问题时，自动跳过所有已处理的日志文件，节省资源
5. **批量处理**：如需批量处理多个日志文件，使用以下命令跳过已处理的：
   ```bash
   ls docs/错误日志/*.log | grep -v '\[已处理\]'
   ```

**回复重点记录（自动执行，不可跳过）**：每次完成用户需求后（代码修改、功能实现、问题修复等），必须**立即**将回复重点记录到当天的修改记录文件中，**不需要等待用户命令**：
1. 定位当天的修改记录文件：`docs/修改记录/{M}月/第N章 修改记录{M}-{D}.md`
2. 在对应模块的标题下（如「宠物模块」），追加本次修改的简要说明
3. 记录格式：`- [功能名称]：简要描述修改内容和实现方式`
4. 若当天修改记录文件不存在，先执行「创建今日修改记录」指令
5. **执行时机**：每次代码修改、功能实现、问题修复完成后，立即更新，不要等用户说"更新修改记录"
6. 此操作为强制执行，不可跳过

**修改记录时间戳规范**：在修改记录文件中，需要为每个用户提问和回复添加时间戳：
1. 用户提问位置：在「**提问：**」后添加时间戳 `[HH:MM]`
2. 回复结果位置：在「**修复：功能名称**」后添加时间戳 `[HH:MM]`
3. 时间格式：24小时制，如 `[08:30]`、`[14:15]`
4. 时间获取方式：使用 `date '+%H:%M'` 命令获取当前时间，不要猜测
5. 匹配优化：由于修改记录文件内容较多，使用用户提问的**前20个字符**作为搜索关键词定位位置，避免全文扫描
6. 示例格式：
   ```
   **提问：** `[08:30]`
   1. 用户的具体问题内容...

   **修复：功能名称** `[08:45]`
   - 问题描述...
   - 修复方案...
   ```

**代码规范（强制执行，不可跳过）**：所有代码修改必须遵守 `docs/coding-standards.md` 中定义的规范，核心要求：
1. **模块化拆分**：每个文件只做一件事，单个文件不超过 300 行
2. **组件化**：复杂逻辑必须抽取为工具函数，禁止在页面内内联超过 30 行的逻辑块
3. **命名约定**：页面文件 kebab-case、组件 PascalCase、工具函数 camelCase
4. **异常捕获**：所有风险代码必须 try/catch，catch 内必须 `console.error` + 中文场景描述 + 原始错误，禁止空 catch 块
5. **日志规范**：`console.error` 用于业务报错，`console.warn` 用于告警降级，禁止 `console.log` 提交到线上，日志必须中文
6. **代码修改后自检**：每次代码修改后，检查文件行数、职责、命名、异常捕获、日志规范
7. **禁止事项**：禁止单文件堆砌功能、禁止空 catch 块、禁止无中文日志、禁止使用 `var` 声明变量

**docs 文件元数据规范（强制执行，不可跳过）**：在 `docs/` 目录下新建文件时（`docs/修改记录/` 目录除外），必须在文件顶部添加 YAML frontmatter 元数据块：
1. **适用范围**：`docs/` 下所有 `.md`、`.txt`、`.log`、`.json` 等文本文件
2. **排除范围**：`docs/修改记录/` 目录下的所有文件、图片文件（`.png`、`.jpg`、`.jpeg`、`.gif` 等）
3. **必须字段**：
   ```yaml
   ---
   description: 文件简要描述（1句话，从文件名或内容首行提取）
   created: YYYY-MM-DD（文件创建日期，使用文件系统时间）
   last_modified: YYYY-MM-DD（最后修改日期，使用文件系统时间）
   ---
   ```
4. **格式要求**：
   - frontmatter 必须是文件最顶部的内容，前面不能有空行
   - `description` 字段：从文件名或内容首行提取，长度 5-80 字符
   - 时间字段：使用 `stat -c %w`（创建时间）和 `stat -c %y`（修改时间）获取，格式 `YYYY-MM-DD`
5. **执行时机**：每次创建新文件后，立即添加 frontmatter，不要等用户提醒
6. **已有文件处理**：如果文件已有 frontmatter，保持不变；如果没有，需要补添加
